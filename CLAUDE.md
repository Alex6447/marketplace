## О проекте

Серверная система, которая по данным о товаре (фото, характеристики, преимущества, стиль бренда) генерирует комплект визуальных карточек для маркетплейсов (Ozon, Wildberries, Яндекс Маркет) с итеративной правкой через текстовую обратную связь менеджера.

**Главный риск и критерий успеха:** сохранение товара без искажений (форма, цвет, детали, логотип, пропорции).

## Технологический стек

- **Backend:** Python 3.12 + FastAPI (тонкий образ); отдельный GPU/CPU worker (тяжёлый образ с torch/CUDA).
- **Очередь:** Redis + Celery (`chain`/`group`/`chord` под многостадийный пайплайн).
- **БД:** PostgreSQL. **Файлы/изображения:** MinIO (S3 API).
- **Frontend:** React + TypeScript + Vite, TanStack Query, shadcn/ui. SPA-дашборд.
- **LLM/Image:** провайдер-абстракции `LLMProvider` / `ImageProvider` — переключение hosted ⇄ local без правки пайплайна.
  - Сохранение товара: editing-модели (Flux.1 Kontext / Gemini 2.5 Flash Image / Qwen-Image-Edit) + композитинг (BiRefNet/SAM2) + ComfyUI (inpaint/ControlNet).
  - Наложение текста: HTML/CSS-шаблоны + Playwright (не нейросеть). Pillow — fallback.
  - Прогресс в UI: SSE (не WebSocket).
- **Тулинг:** `uv` + `ruff` (Python), `pnpm` (frontend).
- **Контейнеризация:** Docker + docker-compose.

## Ключевые архитектурные принципы

- Текст накладывается **отдельным детерминированным этапом**, а не нейросетью.
- Товар берётся с **реального фото** и сохраняется, а не «рисуется заново».
- Пайплайн — набор **изолированных, перезапускаемых стадий** с сохранением артефактов; фидбэк перегенерирует только нужную стадию (контент-адресуемый кэш).
- **Провайдеро-независимость:** работа как на внешних API, так и на локальных моделях (закрытый контур).
- **Стратегия:** API-first для MVP (hosted-модели, без GPU); локальные модели — отдельный этап.

## Окружение и БД

- Конфиг — в `.env` (в гите игнорируется; шаблон — `.env_example`).
- Строка подключения: ключ `DATABASE_URL`, значение `postgresql://postgres:postgres@localhost:5432/marketplace`.
- БД `marketplace` создана на `localhost:5432` (пользователь `postgres`), **пока пустая** (без таблиц/миграций).
- ⚠️ Это локальная dev-БД. Тесты должны использовать отдельную тестовую БД, не `marketplace`.

## Команды

> Раздел дополняется по мере появления кода. Актуально на сейчас (все — из корня репо).

**Python (uv-воркспейс: `apps/api`, `apps/worker`, `packages/shared`):**
- `uv sync` — поставить все пакеты воркспейса + dev-группу (ruff).
- `uv run ruff check .` — линт всего Python-кода.
- `uv run uvicorn marketplace_api.main:app --reload` — запустить тонкий API локально.
- `uv run celery -A marketplace_worker.celery_app:app worker` — запустить воркер.
- `uv sync --package marketplace-api` — окружение только тонкого API.

**Frontend (pnpm-воркспейс):** `pnpm install` (каркас Vite/React — следующий пункт Этапа 0).

**Образы (контекст сборки — корень репо):**
- `docker build -f apps/api/Dockerfile -t marketplace-api .` — тонкий образ API.
- `docker build -f apps/worker/Dockerfile -t marketplace-worker .` — тяжёлый образ воркера.

**Инфраструктура:** `docker compose up -d` (postgres/redis/minio); `docker compose --profile app up -d --build` (+ api/worker).
